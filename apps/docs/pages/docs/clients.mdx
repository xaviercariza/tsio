# Clients

Clients are generated from a transport adapter and a contract.

```ts
import { createClient } from '@tsio/core'

const client = createClient(api, adapter)
```

The generated client mirrors the contract shape and exposes two trees:

```ts
client.actions
client.events
```

## Calling actions

Fire-and-forget actions return `void` or `Promise<void>` depending on the adapter.

```ts
client.actions.presence.updateTypingState({
  chatId: 'chat-1',
  isTyping: true,
})
```

Request-response actions return `TResponse<TData>` or `Promise<TResponse<TData>>`.

```ts
const result = await client.actions.chat.sendMessage({ text: 'Hello' })

if (result.success) {
  result.data.id
} else {
  result.error
}
```

The input type is inferred from the action input schema.

## Listening to events

Events subscribe to server-to-client messages.

```ts
const subscription = client.events.chat.onMessage(message => {
  message.id
  message.text
})

subscription.unsubscribe()
```

The callback payload type is inferred from the event data schema.

## Type-safe boundaries

The generated client prevents common mistakes:

```ts
// @ts-expect-error wrong input shape
client.actions.chat.sendMessage({ wrong: 'field' })

// @ts-expect-error events are not actions
client.actions.chat.onMessage({ id: '1', text: 'Hello' })

// @ts-expect-error actions are not events
client.events.chat.sendMessage(() => {})
```

## Adapter ownership

The client does not know whether it is using Socket.IO or `ws`. That is adapter responsibility.

```ts
const socket = io('http://localhost:3000')
const adapter = socketioClient(socket)
const client = createClient(api, adapter)
```

Keep transport setup close to your app shell and pass the generated client through your preferred dependency mechanism.

In React, a provider is a natural place to create the typed client once.
