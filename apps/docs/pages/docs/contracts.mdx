# Contracts

Contracts are the public protocol for a `tsio` application. They should be small, explicit, and shared by both server and client.

## Define a contract

```ts
import { contract } from '@tsio/core'
import { z } from 'zod'

export const api = contract({
  chat: {
    sendMessage: {
      type: 'action',
      input: z.object({ text: z.string() }),
      response: z.object({ id: z.string(), text: z.string() }),
    },
    onMessage: {
      type: 'event',
      data: z.object({ id: z.string(), text: z.string() }),
    },
  },
})
```

## Action shape

```ts
{
  type: 'action'
  input: z.ZodTypeAny
  response?: z.ZodTypeAny
  options?: {
    validate?: boolean
  }
}
```

Use `response` for request/response actions. Omit `response` for fire-and-forget actions.

## Event shape

```ts
{
  type: 'event'
  data: z.ZodTypeAny
  validate?: boolean
}
```

Events describe server-to-client payloads. Server handlers emit events with `emit`.

## Nested routers

Contracts can be nested to keep large protocols organized:

```ts
const api = contract({
  rooms: {
    members: {
      invite: {
        type: 'action',
        input: InviteMemberSchema,
      },
      onInvited: {
        type: 'event',
        data: InvitedSchema,
      },
    },
  },
})
```

The generated paths are dot-separated:

- `rooms.members.invite`
- `rooms.members.onInvited`

## Best practices

Keep contracts transport-agnostic. A contract should describe business-level protocol messages, not Socket.IO-specific implementation details.

Use domain names for top-level routers:

```ts
chat.sendMessage
presence.updateTypingState
game.movePlayer
```

Avoid transport-flavored names:

```ts
socketEmitMessage
websocketMessageReceived
```

Prefer shared schema names for important domain types:

```ts
const UserSchema = z.object({ ... })
const MessageSchema = z.object({ ... })
```

That makes contracts easier to scan and reuse.
