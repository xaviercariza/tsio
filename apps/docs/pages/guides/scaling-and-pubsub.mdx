# Scaling & Pub/Sub

`tsio` currently focuses on socket transports. That is the right first target because actions and listeners map naturally to a connected client.

However, the contract-first model can evolve toward broader messaging patterns.

## The current model

Today, the core flow is:

```txt
client action -> server handler -> optional response
server handler -> emit listener event -> target client
```

That works well for WebSocket and Socket.IO because the server has a live connection to a specific client.

## Horizontal scaling

When you run multiple Node processes, a socket may be connected to a different process than the one handling an action.

A future backplane adapter could route events between processes:

```txt
process A emits to socket id X
process A publishes through Redis or NATS
process B owns socket id X
process B delivers the listener event
```

That is the most natural first pub/sub extension for `tsio`.

## Supabase or database realtime

Some apps do not need a custom socket server. For example:

```txt
Expo app -> API route -> Supabase
Supabase Realtime -> client update
```

In that architecture, `tsio` should not force WebSockets. A future Supabase Realtime adapter could type listener events over Supabase channels while actions remain HTTP/API-route based.

## Future adapter directions

Possible future adapters:

- `@tsio/http` for request/response actions over `fetch`
- `@tsio/supabase-realtime` for typed realtime subscriptions
- `@tsio/redis-backplane` for multi-process socket delivery
- `@tsio/nats` for service-to-service event protocols

## Design guidance

Do not collapse every transport into a single lowest-common-denominator API.

Different transports have different capabilities:

- targeted client emit
- broadcast
- request/response
- durable topics
- presence
- replay
- rooms

The core contract should remain stable, while adapters declare or document the semantics they support.

## Recommendation

For the next release, keep the stable docs focused on Socket.IO and `ws`.

Document pub/sub and Supabase as future directions, not stable APIs.
